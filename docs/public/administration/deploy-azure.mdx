Azure is a good fit for hosting Fabro when you want Azure-native infrastructure for both the long-running control plane and the workflow sandbox runtime. In this setup, `fabro-server` runs in Azure Container Apps, while workflow sandboxes run as Azure Container Instances provisioned by the server.

This guide documents the supported production path for this repo:

- one-time Terraform bootstrap for a GitHub Actions user-assigned managed identity and remote state
- one manual Terraform apply for the shared Azure environment, with no `-target` workaround
- manual GitHub Actions deploys that build, push, apply immutable ACR images, and verify a no-op follow-up plan
- runtime configuration completed through the web install wizard and persisted on `/storage`

<Note>
  Fabro currently assumes a single active writer on `/storage`. Keep the Azure
  Container App at exactly one replica.
</Note>

## Deployment model

- **Bootstrap root:** `terraform/bootstrap/github_actions`
- **Steady-state environment:** `terraform/environments/sandbox`
- **Server runtime:** `fabro-server` in Azure Container Apps
- **Workflow sandboxes:** Azure Container Instances
- **Persistent app state:** Azure Files mounted at `/storage`
- **Image registry:** Azure Container Registry for both `fabro-server` and `fabro-azure-sandbox-base`

## Prerequisites

- An Azure subscription
- Terraform `>= 1.8.0`
- The latest [`az`](https://learn.microsoft.com/cli/azure/install-azure-cli) CLI
- Docker with `buildx`

## Supported greenfield sequence

- create a GitHub environment named `production`
- copy bootstrap outputs into that environment
- add the environment-specific Azure naming variables
- create the GitHub App manually and store its values in the GitHub `production` environment before the first browser-login-ready deploy
- run the first `terraform/environments/sandbox` apply with `fabro_server_enabled = false`
- trigger `.github/workflows/deploy-azure.yml`
- let CI complete install, persist `FABRO_DEPLOY_DEV_TOKEN`, and continue authenticated checks in the same run when `/health` reports install mode and the GitHub App values are present
- use the manual fallback only when the GitHub App values are missing

## 1. Bootstrap GitHub Actions managed identity and remote state

Sign in to Azure first:

```bash
az login
az extension add -n containerapp --upgrade
az provider register --namespace Microsoft.App
az provider register --namespace Microsoft.OperationalInsights
```

Copy the bootstrap example and fill in the real Azure names for this deployment:

```bash
cp terraform/bootstrap/github_actions/terraform.tfvars.example \
  terraform/bootstrap/github_actions/terraform.tfvars
```

Then apply the bootstrap root:

```bash
terraform -chdir=terraform/bootstrap/github_actions init
terraform -chdir=terraform/bootstrap/github_actions apply
```

This root creates the backend resource group, storage account, storage container,
the GitHub Actions user-assigned managed identity, the GitHub OIDC federated
credential, and the backend storage access Terraform needs for CI.

The bootstrap root grants the CI identity all backend storage permissions the current
`azurerm` backend flow needs: `Storage Blob Data Contributor`, `Reader`, and
`Storage Account Key Operator Service Role` on the backend storage account.

Save these Terraform outputs into the GitHub `production` environment as variables:

- `AZURE_CLIENT_ID`
- `AZURE_TENANT_ID`
- `AZURE_SUBSCRIPTION_ID`
- `AZURE_GITHUB_ACTIONS_PRINCIPAL_ID`
- `TF_BACKEND_RESOURCE_GROUP`
- `TF_BACKEND_STORAGE_ACCOUNT`
- `TF_BACKEND_CONTAINER`
- `TF_BACKEND_KEY`

The bootstrap root is intentionally one-time only. After this, the supported environment is `terraform/environments/sandbox`.

## 2. Create the shared Azure environment once

Copy the sandbox example:

```bash
cp terraform/environments/sandbox/terraform.tfvars.example \
  terraform/environments/sandbox/terraform.tfvars
```

Initialize the remote backend using the values you saved from bootstrap:

```bash
terraform -chdir=terraform/environments/sandbox init \
  -backend-config="resource_group_name=<TF_BACKEND_RESOURCE_GROUP>" \
  -backend-config="storage_account_name=<TF_BACKEND_STORAGE_ACCOUNT>" \
  -backend-config="container_name=<TF_BACKEND_CONTAINER>" \
  -backend-config="key=<TF_BACKEND_KEY>"
```

Create the GitHub `production` environment before continuing if it does not already
exist. Then add the bootstrap outputs as environment variables, followed by the
environment-specific Azure naming variables used by `.github/workflows/deploy-azure.yml`.

For the first apply, keep `fabro_server_enabled = false` and set `github_actions_principal_id` to the bootstrap-created GitHub Actions managed identity principal ID so Terraform can grant CI access during greenfield bring-up. Run a normal apply with no `-target` flags:

```bash
terraform -chdir=terraform/environments/sandbox apply
```

The first `terraform/environments/sandbox apply` is a required operator step even in the
managed-identity model. It creates the CI role assignments in the sandbox environment before
GitHub Actions takes over the steady-state deploy path.

Keep that first manual apply disabled. The server turns on later in CI: `.github/workflows/deploy-azure.yml` exports `TF_VAR_fabro_server_enabled=true` and runs `terraform apply` with the real immutable `fabro-server` image for the Azure-ready ref being deployed.

That apply provisions the resource group, networking, storage, ACR, a privileged server identity for `fabro-server`, a separate low-privilege sandbox-pull identity for Azure sandbox image pulls, the role assignment that lets the server attach that sandbox-pull identity to ACI groups, the Container Apps environment, and the GitHub Actions access needed for steady-state CI deploys.

Record these Terraform outputs for the install flow:

- `resource_group_name`
- `aci_subnet_id`
- `acr_login_server`
- `storage_account_name`
- `blob_data_container_name`
- `acr_identity_resource_id`

Also record the Azure subscription and location values used for this environment:

- `AZURE_SUBSCRIPTION_ID`
- `AZURE_LOCATION`

The checked-in Terraform defaults currently assume `fabro-server` runs on Azure Container Apps with
`2 CPU / 4Gi`. That is the validated shape for the Azure deploy path in this repo.
## 3. Configure the GitHub `production` environment

Add the Azure and Terraform backend variables from bootstrap, then add the sandbox environment variables consumed by `.github/workflows/deploy-azure.yml`:

- `AZURE_LOCATION`
- `AZURE_RESOURCE_GROUP_NAME`
- `AZURE_VNET_NAME`
- `AZURE_VNET_CIDR`
- `AZURE_ACA_SUBNET_NAME`
- `AZURE_ACA_SUBNET_CIDR`
- `AZURE_ACI_SUBNET_NAME`
- `AZURE_ACI_SUBNET_CIDR`
- `AZURE_STORAGE_ACCOUNT_NAME`
- `AZURE_SERVER_STORAGE_SHARE_NAME`
- `AZURE_ACR_NAME`
- `AZURE_IDENTITY_NAME`
- `AZURE_CONTAINER_APPS_ENVIRONMENT_NAME`
- `AZURE_FABRO_SERVER_NAME`

Before the first browser-login-ready deploy, create the GitHub App manually and store its values in the GitHub `production` environment.

Required environment variables:

- `FABRO_DEPLOY_GITHUB_APP_ID`
- `FABRO_DEPLOY_GITHUB_APP_CLIENT_ID`
- `FABRO_DEPLOY_GITHUB_APP_SLUG`
- `FABRO_DEPLOY_GITHUB_ALLOWED_USERNAME`

Required environment secrets:

- `FABRO_DEPLOY_GITHUB_APP_CLIENT_SECRET`
- `FABRO_DEPLOY_GITHUB_APP_PRIVATE_KEY`
- `FABRO_DEPLOY_GITHUB_TOKEN`

Optional environment secret:

- `FABRO_DEPLOY_GITHUB_APP_WEBHOOK_SECRET`

`FABRO_DEPLOY_GITHUB_TOKEN` is no longer sent to the server as the installed GitHub credential. It remains required only so the deploy workflow can validate repository access and persist `FABRO_DEPLOY_DEV_TOKEN` back into the GitHub `production` environment after a successful install.

After the first successful install, add one secret if the workflow did not already persist it:

- `FABRO_DEPLOY_DEV_TOKEN`

That token is used only for the authenticated API check and the tiny post-deploy smoke workflow.

In the server's `settings.toml`, the GitHub App bootstrap path should leave both `"dev-token"` and `"github"` in `[server.auth].methods` so browser OAuth and operator dev-token access stay available together.

## 4. Run the first CI deploy

Trigger the manual workflow from GitHub using the Azure-ready branch or ref you want to deploy:

- workflow: `.github/workflows/deploy-azure.yml`
- environment: `production`
- input: the Git ref to deploy

For forks that keep `main` as an upstream mirror, do not assume `main` contains the Azure deploy workflow. Run the manual deploy from the Azure-ready branch or tag that carries `.github/workflows/deploy-azure.yml` and the matching infrastructure changes.

The deploy workflow does seven things:

1. Logs into Azure with GitHub OIDC.
2. Builds and pushes `fabro-server:<deploy-id>` and `fabro-azure-sandbox-base:<deploy-id>` to ACR.
3. Runs `terraform apply` with the exact immutable `fabro-server` image ref.
4. Runs a follow-up `terraform plan -detailed-exitcode` to prove the apply reached steady state.
5. Waits for the new Container App revision to report ready before hitting the live URL.
6. Reads `/health`, auto-completes install when the server reports install mode and the GitHub App values are present, persists `FABRO_DEPLOY_DEV_TOKEN`, waits for the configured auth state after install, and then runs the authenticated checks in the same run. When `/health` already returns plain `{"status":"ok"}`, install is skipped and the workflow continues directly, but authenticated validation still runs only if a dev token is already available.
7. Publishes a GitHub Actions step summary with the ref, deploy ID, image refs, ready revision, and whether the authenticated checks ran.

Deploy IDs use `<utc-timestamp>-<short-sha>`, for example `20260428T193500Z-a1b2c3d`.

Treat the `/health` response as the source of truth for what happens next in CI. The same `deploy-azure.yml` run is intentionally idempotent across first deploys and later redeploys.

If `/health` returns `{"status":"ok","mode":"install"}` and the GitHub App values are present, the workflow completes install headlessly, persists `FABRO_DEPLOY_DEV_TOKEN`, waits for the configured state, and continues with authenticated validation in the same run.

If `/health` returns install mode and the GitHub App values are missing, the workflow still pre-fills the server, Azure, object-store, sandbox, and LLM install steps, then stops at the GitHub App linkage handoff point. Complete the remaining GitHub step manually in the browser, store the emitted `FABRO_DEPLOY_DEV_TOKEN`, and rerun `.github/workflows/deploy-azure.yml`.

If `/health` returns plain `{"status":"ok"}`, the workflow skips install and continues directly. The authenticated checks run only when `FABRO_DEPLOY_DEV_TOKEN` is already available; otherwise those authenticated checks remain skipped.

## 5. Install completion

Once the Container App is reachable, let the deploy workflow handle install when `/health` reports install mode. In the supported path, CI can complete install headlessly with the pre-existing GitHub App values, persist `FABRO_DEPLOY_DEV_TOKEN`, wait for the configured state, and continue with the authenticated checks in the same run.

If the deploy workflow already observed `/health` return `{"status":"ok","mode":"install"}`, treat that as a
signal to finish install immediately before attempting any authenticated checks. If `/health` already returns plain
`{"status":"ok"}`, the deployment is out of install mode and CI skips install and continues directly. Authenticated
validation only runs when a dev token is already available; otherwise those authenticated checks remain skipped.

For day-two deploys after bootstrap, rerunning `.github/workflows/deploy-azure.yml` from the current Azure-ready branch or ref is the normal way to ship a new version. The workflow stays on the same idempotent path: `/health` decides whether install still needs to run, and healthy configured deployments go straight to validation.

The important runtime model is:

- the Container App mounts `/storage`
- the install wizard writes runtime configuration there
- the install wizard's `Azure` step should use the Terraform output `acr_identity_resource_id`
- on the sandbox step, choose `Azure Container Instances` so the wizard writes `provider = "azure"` and reuses the Azure subnet, registry, and managed-identity settings collected on the Azure step
- the sandbox-attached identity for ACR pulls is the dedicated low-privilege sandbox-pull identity, not the privileged `fabro-server` identity
- Azure Blob is the durable data plane for artifacts, SlateDB, and archived raw run logs
- later deploys reuse the same persisted config and Blob-backed runtime data

The server should come up in install mode without relying on install-time `SESSION_SECRET`, `FABRO_DEV_TOKEN`, or `FABRO_AZURE_*` worker env injection from Terraform.

For the supported workflow-managed install path, the bootstrap automation uses a hard-coded dummy OpenAI API key only to
leave install mode. Replace the resulting LLM provider configuration manually after the deployment succeeds.

If you need the manual fallback instead, open the web install wizard after the Container App is reachable. The workflow is still idempotent across first deploys and later redeploys, but this first workflow run intentionally fails after bringing up the app when `/health` reports install mode and the GitHub App values are missing. That failure is the handoff point for the operator to finish GitHub App linkage manually and then rerun `.github/workflows/deploy-azure.yml` for the authenticated validation path.

For that manual Azure install path, capture or derive the same values the workflow sends to `/install/azure` and `/install/object-store`:

- `subscription_id` from `AZURE_SUBSCRIPTION_ID`
- `resource_group` from the Terraform `resource_group_name` output
- `location` from `AZURE_LOCATION`
- `aci_subnet_id` from the Terraform `aci_subnet_id` output
- `acr_login_server` from the Terraform `acr_login_server` output
- `acr_identity_resource_id` from the Terraform `acr_identity_resource_id` output
- `storage_account_name`
- `blob_data_container_name`

After the manual install succeeds:

1. Save the dev token shown by the wizard.
2. Store it in the GitHub `production` environment as `FABRO_DEPLOY_DEV_TOKEN`.
3. Use the same workflow for every steady-state deploy.
4. Rerun `.github/workflows/deploy-azure.yml` so the authenticated API check and Azure smoke workflow execute against the configured deployment.

## 6. Shared sandbox image and workflow-specific images

The deploy workflow publishes a shared Azure sandbox base image to ACR:

- `fabro-azure-sandbox-base:<deploy-id>`

The checked-in smoke workflow uses a placeholder image ref:

```toml title=".fabro/workflows/azure-deploy-smoke/workflow.toml"
[run.sandbox.azure]
image = "example.azurecr.io/fabro-azure-sandbox-base:example"
```

CI replaces that placeholder with the just-built immutable image before running the smoke workflow.

Workflow-specific sandbox images still live in the workflow repositories that use them. Those refs are updated manually by workflow owners.

## 7. Post-deploy validation

Steady-state production deploys should validate all three of these checks:

1. `GET /health`
2. authenticated `GET /api/v1/models`
3. run `.fabro/workflows/azure-deploy-smoke` against the deployed server

Equivalent manual checks:

```bash
curl -fsSL "$FABRO_SERVER_URL/health"

curl -fsSL \
  -H "Authorization: Bearer $FABRO_DEPLOY_DEV_TOKEN" \
  "$FABRO_SERVER_URL/api/v1/models"

rm -rf /tmp/azure-deploy-smoke
mkdir -p /tmp/azure-deploy-smoke
# Preserve both workflow.toml and workflow.fabro in the temp bundle.
cp -R .fabro/workflows/azure-deploy-smoke/. /tmp/azure-deploy-smoke/

sed -i.bak \
  "s|image = \".*\"|image = \"$ACR_SERVER/fabro-azure-sandbox-base:$DEPLOY_ID\"|" \
  /tmp/azure-deploy-smoke/workflow.toml

rm -f /tmp/azure-deploy-smoke/workflow.toml.bak

export FABRO_DEV_TOKEN="$FABRO_DEPLOY_DEV_TOKEN"
# or run `fabro auth login --server "$FABRO_SERVER_URL/api/v1"` first

fabro run /tmp/azure-deploy-smoke/workflow.toml \
  --server "$FABRO_SERVER_URL/api/v1"
```

If the smoke workflow succeeds, the deployed server has proven that it can launch an Azure sandbox using the shared base image.

If `FABRO_DEPLOY_DEV_TOKEN` is not set yet on an already-configured deployment, the workflow still passes `/health` and
skips the authenticated API check and smoke workflow until a reusable dev token is available. If the server is still in
install mode instead, finish install through the supported auto-install path or the manual fallback, store the emitted
token as `FABRO_DEPLOY_DEV_TOKEN`, and then run the deploy workflow again to exercise the authenticated checks.

## 8. Rollback

Rollbacks use the same workflow. Re-run `.github/workflows/deploy-azure.yml` with the last known-good Git ref.
That rebuilds immutable images for that ref, repoints Terraform at the older server image, and waits for the
replacement Container App revision before the smoke checks run.

For an operational rollback:

1. Pick the last deploy whose `/health`, authenticated `/api/v1/models`, and smoke workflow all passed.
2. Trigger the manual deploy workflow with that Git ref.
3. Confirm the new workflow run reports a healthy ready revision and passes the post-deploy checks.

Because `/storage` and the Blob-backed runtime data stay in place, this rollback path is intended for code or image
regressions, not for reverting install-wizard state changes.

## 9. CI validation checklist

Before relying on CI deploys, verify all of the following:

- the bootstrap root has already created the GitHub Actions OIDC identity and remote backend
- `terraform/environments/sandbox` has been applied once manually with `fabro_server_enabled = false`
- the GitHub `production` environment contains every Azure and Terraform variable listed in this guide
- the GitHub App values are present for headless bootstrap, or manual GitHub App linkage has already completed
- `FABRO_DEPLOY_DEV_TOKEN` is stored in the GitHub `production` environment
- the Container App is still configured for exactly one replica
- the deploy workflow can pass `GET /health`, authenticated `GET /api/v1/models`, and the Azure smoke workflow

For the very first CI deploy, the last three checklist items become the follow-up gate after install is complete rather
than a prerequisite for bringing the app up.

## 10. Fork and resource-group reuse

The supported production shape is one GitHub repository environment driving one steady-state Azure environment.

If you are testing from a fork, create a separate bootstrap state key, GitHub environment, and Azure resource group for
that fork. Do not point two repositories at the same production backend and resource group unless they are intentionally
operating the exact same deployment.

If you need to reuse an existing Azure resource group, keep the Terraform backend and all GitHub environment variables
aligned with the names already provisioned there. The deploy workflow only works when the GitHub environment, the
Terraform state, and the live Azure resource group all describe the same environment.

## 11. Troubleshooting and caveats

- **Single replica only.** Keep the Container App at one replica.
- **The first production deploy is two-phase.** Bootstrap and run the shared environment apply once before expecting CI to deploy the server.
- **Remote state is required.** Local Terraform state is no longer the supported production path for this environment.
- **ACI quotas are easy to exhaust.** Trial subscriptions often need stale `fabro-*` container groups cleaned up before retries.
- **CI smoke checks need a dev token.** The headless GitHub App path persists one automatically when the GitHub App values are present; otherwise finish GitHub App linkage manually, store `FABRO_DEPLOY_DEV_TOKEN`, and rerun the workflow.
- **The smoke workflow validation command is server-backed.** `fabro validate` currently uses server preflight, so it needs a configured server target rather than an unconfigured local machine.
- **Workflow-specific images are still manual.** This repo only automates the shared deployment substrate.

To inspect or clean up stale Azure sandboxes:

```bash
az container list \
  --resource-group "$AZURE_RESOURCE_GROUP_NAME" \
  -o table

az container delete \
  --resource-group "$AZURE_RESOURCE_GROUP_NAME" \
  --name <stale-fabro-container-group> \
  --yes
```
